Architecture & System Design
Class and Type Relationships
The Square class encapsulates all entity state and behavior. Type alias Color simplifies RGB tuples used throughout. Main game loop coordinates square updates and rendering.
Square.update() - Frame Update Algorithm
For each frame, every square undergoes velocity perturbation, position update, collision handling, neighbor detection via spatial grid, and behavior steering (flee/chase/wander).
Function Call Relationships
main() initializes pygame and enters the game loop. Each frame calls update() on all squares, then draw() for rendering. Square.update() internally performs spatial partitioning and behavior logic.
Runtime Sequence: One Frame Execution
Event processing → all squares updated (each builds grid and steers) → dead squares replaced → screen cleared and redrawn → display flipped and clock ticked.
Data Flow: Behavior Decision Pipeline
Position and size data flows through spatial grid lookup, enabling threat and prey detection. Threat/prey data drives steering decisions that modify velocity, which in turn updates position.
Patterns & Best Practices

✅ Good Patterns

✅ Pattern 1
Spatial Grid Optimization (O(k) vs O(n²))
Basics
Instead of checking all n squares for neighbors, partition the 800×600 space into 100-pixel cells. Each square looks only at its 3×3 neighborhood of cells, finding ~k neighbors in O(1) amortized time.
Engineering Insight
The grid dict maps (cell_x, cell_y) to lists of squares. When querying neighbors, iterate only over the ~9 cells around the square's position. Worst case is O(n) if all squares cluster, but typical case is O(n·c) where c ≈ constant.
Architecture Insight
This is a classic **space partitioning** technique used in graphics engines, physics simulators, and real-time games. It scales to thousands of entities with minimal frame-time impact. The trade-off: using O(n·c) memory for the grid.
✅ Pattern 2
Normalized Direction Vectors for Stable Steering
Basics
When fleeing or chasing, the code normalizes the direction vector (divide by distance) before applying speed. This ensures velocity magnitude stays constant, preventing wild acceleration spikes.
Engineering Insight
In fleeing: `flee_dir = -dv / distance` produces a unit vector. Multiplying by `current_speed` yields a stable velocity. In chasing: blend `current_velocity` with chase direction using `CHASE_STRENGTH` (0.2), creating smooth steering without jerking.
Architecture Insight
This is **vector normalization** in game AI: separating direction (unit vector) from magnitude (speed). Allows independent control of "where to go" vs. "how fast." Critical for responsive, balanced behavior tuning.
✅ Pattern 3
Separation of Concerns: Update, Draw, Loop
Basics
Square.update() handles logic (physics, behavior). Square.draw() handles rendering. main() orchestrates the loop. Each method has one clear responsibility.
Engineering Insight
If you want to change AI behavior, edit update(). If you want visual style, edit draw(). If you want frame rate or event handling, edit main(). Changes are isolated and don't ripple through the system.
Architecture Insight
This mirrors the **Model-View-Controller (MVC)** pattern: Square is Model (data + logic), draw() is View, main() is Controller. Scales to complex projects when applied consistently.

⚠️ Potential Issues

⚠️ Issue 1
Inefficient Grid Recalculation Every Frame
Basics
Inside Square.update(), the spatial grid is rebuilt for ALL squares every single frame. This is unnecessary: each square only needs to query neighbors once per update cycle.
Engineering Insight
Current: O(n) grid rebuilds × 60 fps × 5 squares = regenerated 300 times/sec. Better: build grid once per frame in main(), pass to all square.update() calls. Reduces dict overhead and redundant lookups.
Architecture Insight
This violates the **Don't Repeat Yourself (DRY)** principle. Shared data structures (grid) should live in a higher scope (main loop) and injected into methods, not reconstructed in each method.
⚠️ Issue 2
Potential Division by Zero in Vector Normalization
Basics
If a threat or prey square occupies the exact same center as the current square, distance becomes 0. Dividing by zero produces NaN, corrupting velocity.
Engineering Insight
Add a guard: `if distance > 0.01: normalize_and_steer()`. Or clamp distance to a minimum epsilon value. Without this, overlapping squares can freeze or behave erratically.
Architecture Insight
**Defensive programming**: anticipate edge cases (overlaps, zero vectors, NaN) and handle them explicitly. In game dev, these edge cases cause hard-to-debug glitches that surface only under specific conditions.

Performance Analysis
🚀 Performance Wins
Win 1
Spatial Grid Reduces Neighbor Lookup from O(n²) to O(k)
By partitioning space into 100×100 cells, neighbor queries run in constant (or near-constant) time. With 5 squares and FLEE_RADIUS=50, each square checks ~4 neighbors instead of 4. At scale (100+ squares), this saves thousands of distance calculations per frame.
Win 2
Normalized Vectors Eliminate Vector Recalculation
Fleeing and chasing both normalize direction before scaling by speed. This avoids redundant distance calculations and keeps velocity magnitude stable—no expensive magnitude recomputation needed.
🐢 Performance Risks
Risk 1
Grid Rebuilt Per Square Per Frame
Grid is reconstructed inside each square's update() call. Current: n squares × rebuilding full grid = O(n) grid reconstructions per frame. Better: build once in main(), pass to update(). At 1000 squares, this could add 10ms+ of overhead.
Risk 2
Redundant Distance Calculations (Flee/Chase)
Once threat/prey are found, distance is recalculated a second time (for normalization). Store it as `best_threat_distance` and reuse. Saves ~2 sqrt() calls per square per frame—adds up at scale.
3 / 6
Moderate Performance: Spatial grid is a huge win, but redundant grid rebuilds and distance recalculations leak efficiency. At 5 squares, imperceptible. At 100+, noticeable frame lag.

Architecture Overview

A classic game loop with a spatial optimization layer: initialize once, then repeatedly process input, update state using spatial grid for neighbor lookup, and render output at 60 FPS.

Key Function Class Walkthrough
  • Square.__init__: Initialize per-object state: size, spawn point, velocity, color, and lifespan countdown.
  • Square.update: Apply random drift, boundary bounce, and fleeing using spatial grid lookup (O(k) instead of O(n²)).
  • Square.draw: Render square with slight jitter for hand-drawn animation effect.
  • main: Orchestrate pygame lifecycle, frame loop phases, dead square replacement, and cleanup.
  • Spatial Grid (new): Build dict[tuple[int, int], list[Square]] each frame to partition space into cells, enabling fast neighbor queries.

Potential Issues
⚠️Grid Rebuild Is O(n) Per Frame
Although the grid reduces flee lookups from O(n²) to O(k), the grid is rebuilt from scratch each frame. This is O(n) work before any flee queries happen.
Basics:At 30 squares, rebuilding is cheap. At 300+ squares, grid construction becomes the bottleneck.
Engineering Insight:Incremental structures (QuadTree, BVH) or temporal coherence can avoid full rebuilds.
Architecture Insight:Document scaling limits and plan for refactors as object count rises.
Consider a QuadTree or hierarchical spatial grid for production scaling.
⚠️Flee Velocity Calculation Uses Speed Incorrectly
The flee code computes a normalize factor using current velocity magnitude, not a pure direction vector. This can create unstable or oversteer flee behavior.
Basics:Direction and speed are different concepts; should be computed separately.
Engineering Insight:Velocity steering usually normalizes the direction vector first, then scales by desired speed.
Architecture Insight:Clear separation of intent (direction) from magnitude (speed) makes behavior predictable and tunable.
Compute flee direction from threat positions alone, then scale by desired speed independently.

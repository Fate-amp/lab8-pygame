Lab 8 Pygame animation project with spatial grid optimization for fleeing behavior. Focus: main.py. Audience: first-year CS students learning game loops, spatial indexing, and performance optimization.

Architecture Overview

A classic game loop with a spatial optimization layer: initialize once, then repeatedly process input, update state using spatial grid for neighbor lookup, and render output at 60 FPS.

Key Function Class Walkthrough
  • Square.__init__: Initialize per-object state: size, spawn point, velocity, color, and lifespan countdown.
  • Square.update: Apply random drift, boundary bounce, and fleeing using spatial grid lookup (O(k) instead of O(n²)).
  • Square.draw: Render square with slight jitter for hand-drawn animation effect.
  • main: Orchestrate pygame lifecycle, frame loop phases, dead square replacement, and cleanup.
  • Spatial Grid (new): Build dict[tuple[int, int], list[Square]] each frame to partition space into cells, enabling fast neighbor queries.
Function Call Graph
Main entry point reaches the game loop; each frame calls update and draw on every Square object. The spatial grid is internal to update.
Spatial Grid Optimization Flow
Build a 2D cell grid by placing each square in all cells it occupies (handles variable size). Query a 3×3 neighborhood for threats instead of scanning all squares.

Data Flow: Velocity state
Velocity starts from size defaults, is randomly drifted, clamped for stability, used for movement, and redirected by flee steering.
Data Flow: Lifespan state
Birth time recorded at spawn; remaining life decrements each frame; square respawns when remaining life drops below threshold.

Good Patterns
✅Spatial Grid Optimization
The flee behavior uses a grid-based neighbor lookup, reducing complexity from O(n²) to O(k) where k is the number of squares in nearby cells.
Basics:Group nearby squares into cells; only check cells in a 3×3 region instead of all squares.
Engineering Insight:A spatial index is a fundamental technique for scaling simulations from 30 objects to thousands.
Architecture Insight:This design pattern enables extensibility: add collision detection, damage zones, or attract behaviors without O(n²) cost.

Engineering Insight:Avoids drawing half-updated state; improves reproducibility and debugging.
Architecture Insight:Matches game engine conventions and enables future layers: physics, AI, networking.

Potential Issues
⚠️Grid Rebuild Is O(n) Per Frame
Although the grid reduces flee lookups from O(n²) to O(k), the grid is rebuilt from scratch each frame. This is O(n) work before any flee queries happen.
Basics:At 30 squares, rebuilding is cheap. At 300+ squares, grid construction becomes the bottleneck.
Engineering Insight:Incremental structures (QuadTree, BVH) or temporal coherence can avoid full rebuilds.
Architecture Insight:Document scaling limits and plan for refactors as object count rises.
Consider a QuadTree or hierarchical spatial grid for production scaling.
⚠️Flee Velocity Calculation Uses Speed Incorrectly
The flee code computes a normalize factor using current velocity magnitude, not a pure direction vector. This can create unstable or oversteer flee behavior.
Basics:Direction and speed are different concepts; should be computed separately.
Engineering Insight:Velocity steering usually normalizes the direction vector first, then scales by desired speed.
Architecture Insight:Clear separation of intent (direction) from magnitude (speed) makes behavior predictable and tunable.
Compute flee direction from threat positions alone, then scale by desired speed independently.

Wins
Spatial Grid (O(k) vs O(n²))
Flee behavior is now O(k) where k is neighbors per cell, not O(n²) pairwise checks. Scales to hundreds of objects without flee bottleneck.
Frame Rate Cap Is Explicit
clock.tick(FPS) prevents unbounded CPU and keeps animation stable across machines.
Compact Per-Object State
Each square stores only scalar floats; low memory footprint and cache-friendly layout at scale.
Risks
Grid Rebuild Every Frame (O(n))
Spatial grid is constructed from scratch each frame. While O(n) is dominated by O(k) lookups now, at very high object counts or with smaller CELL_SIZE, rebuilds can matter.
Manual Distance Arithmetic
Square root is called in inner loop for distance; comparing squared distances avoids sqrt cost entirely.
No Early-Exit Culling
Once in neighboring cells, every pair is checked. Could skip if distance > FLEE_RADIUS before expensive vector math.
5 / 6 Verdict: Spatial grid provides strong scaling. Grid rebuild (O(n)) is now the secondary bottleneck; optimize if object count exceeds ~1000.

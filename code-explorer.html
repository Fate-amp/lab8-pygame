main.py

Lab 8 Pygame animation project. Focus file: main.py. Audience: first year CS students learning loops, vectors, and architecture basics.

Architecture Overview

The program follows a classic game loop architecture: initialize once, then repeatedly process input, update state, and render output at a capped frame rate.

Key Function Class Walkthrough
  • Square.__init__ seeds per object state: size, spawn point, velocity, color, and lifespan timers.
  • Square.update applies random drift, boundary bounce, flee steering, and life countdown.
  • Square.draw renders each square with tiny jitter for a sketch style motion feel.
  • main orchestrates startup, frame loop phases, replacement of expired squares, and shutdown.

Data Flow: squares list
The squares list is created once and then reused every frame for updates and rendering, making it the central shared game state.
Data Flow: velocity state
Velocity begins from size-based defaults, is perturbed randomly, clamped for stability, and then reused for movement and flee steering.
Data Flow: position state
Position updates each frame from velocity, then receives boundary correction before being consumed by draw output.

Good Patterns
✅Constants Grouped At Top
Global settings like FPS, sizes, and speed bounds are easy to discover and tune without hunting through logic blocks.
Basics:Students can quickly see which numbers control behavior.
Engineering Insight:Centralized constants reduce accidental drift between update and draw logic.
Architecture Insight:Configuration is decoupled from behavior and scales well when adding game modes.
Related reading: Parameter Object and config style
✅Class Encapsulates Square State
The Square class stores position, velocity, size, color, and behavior together, reducing scattered state handling.
Basics:One object equals one square, so reasoning is straightforward.
Engineering Insight:Encapsulation allows adding new behaviors without rewriting the main loop.
Architecture Insight:The loop orchestrates objects while each object owns its own rules.
Related reading: Class responsibilities
✅Update Draw Separation
State updates happen before rendering, which keeps frame output deterministic for a fixed state snapshot.
Basics:Compute first, then paint the frame.
Engineering Insight:This avoids reading half-updated state while drawing.
Architecture Insight:Matches common game engine phase pipelines and supports future optimization passes.
Related reading: Game loop model
Potential Issues
⚠️Tight Coupling To Globals
Square methods directly rely on global WIDTH, HEIGHT, and speed constants, which makes reuse in another scene harder.
Basics:Objects depend on values that live outside the class.
Engineering Insight:Unit tests need global setup to run object behavior correctly.
Architecture Insight:Injecting config into constructor enables scene level composition.
Move window bounds and speed limits into a config object passed to Square.
⚠️Repeated Magic Thresholds
Values like flee radius 60 and background color tuple appear inline, making intent less explicit and tuning less centralized.
Basics:Hidden numbers are harder to remember later.
Engineering Insight:Gameplay balancing becomes fragile if values are spread across methods.
Architecture Insight:Single source constants make behavior policies transparent.
Define named constants for flee distance and frame background color.

Type hints are strong for a beginner game project: key signatures, aliases, and container types are present, with minor consistency gaps in constant declarations.

Strengths

  • Parameter annotations exist where they matter most, including Sequence[Square] and pygame.Surface.
  • Return annotations are consistent across __init__, update, draw, and main.
  • Concrete generics and aliases are used: list[Square], tuple[int, int, int], and Color.

Gaps

  • Top level constants are not fully annotated, so static tools infer their types indirectly.
  • A small typed config object for gameplay constants would improve consistency across modules.
